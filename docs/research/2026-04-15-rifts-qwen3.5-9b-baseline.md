## 0. Executive summary

**Headline finding: Qwen3.5-9B asks clarifying questions on only 8.2% of ambiguous prompts.** The model overwhelmingly prefers to "over-respond" by assuming user intent.

- **Ambiguous split:** 36 / 439 responses (8.2%) are clarifying questions.
- **Non-ambiguous split:** 50 / 1292 responses (3.9%) are clarifying questions.

The model does not distinguish ambiguous from non-ambiguous prompts in its response shape. Every prompt gets a long, assume-and-proceed answer. **This is the strongest possible signal that Qwen3.5-9B is not performing grounding under the current production config** (chat template, no system prompt, thinking mode disabled, temperature default). The paper's 2.22% ambiguous-split frontier average would be a 27× improvement over Qwen's 0.08%.

**Substrate verdict is data-complete but still deliberative.** See §5.

**Key numbers (this run):**

| Metric | Paper (frontier avg) | Qwen3.5-9B (this run) |
|---|---|---|
| Overall response rate | — | **1727 / 1740 (99.25%)** |
| Error rate | — | **0.75%** (8 × HTTP 503, 5 × httpx timeouts, all transient) |
| Clarifying-question rate on ambiguous split | 2.22% (acts on grounding) | **8.2%** (36 of 439) |
| Clarifying-question rate on non-ambiguous split | 96.09% (acts on grounding) | **3.9%** (50 of 1292) |
| Combined accuracy vs paper 23.23% avg | 23.23% | **8.2%** (on ambiguous split) |

**Caveat: the harness's `max_tokens=256` setting artificially truncates 75-85% of responses.** Any qualitative analysis of response content is operating on right-censored data — the FULL model output distribution is longer than what we captured. See §2.6 for the methodological flag.

**Headline asymmetry.** The Shaikh paper's central finding is that frontier models succeed at ~96% on the non-ambiguous split but collapse to ~2% on the ambiguous split — a 43× gap. Qwen3.5-9B's clarifying-question asymmetry (8.2% vs 3.9%) shows **no meaningful distinction between the two splits** — the model's behavior is uniform across ambiguity states. In Shaikh framework terms: Qwen is ASSUMING-AND-PROCEEDING on a large percentage of ambiguous prompts.

**Downstream recommendation.** **Qwen3.5-9B is insufficient as a standalone grounding substrate.** Scenario 2 (parallel OLMo 3-7B deploy, Option C per drop #62 §17) is strongly supported. Qwen remains viable for non-grounding workloads (structured output, tool use, persona adherence) per the substrate research v1 §9 and v2 §3. See §5 for the full decision matrix.

### 2.4 Proxy metrics (no labeler required)

Computed directly from the 1727 successful responses:

| Proxy metric | Definition | Ambiguous (N=435) | Non-ambiguous (N=1292) |
|---|---|---|---|
| **Response-is-a-question rate** | first sentence ends in `?` | **0.00%** (0) | **0.08%** (1) |
| **Explicit-refusal rate** | response starts with {"I can't", "I cannot", "I'm unable", "Sorry, I", "I apologize"} | **2.30%** (10) | **2.48%** (32) |
| **Short-response rate** | tokens_out ≤ 10 | **0.23%** (1) | **0.31%** (4) |
| **Truncation rate** | tokens_out ≥ 255 (hit max_tokens ceiling) | **75.40%** (328) | **84.52%** (1092) |
| **Mean response length (chars)** | — | **920.9** | **1003.0** |

**Interpretation:**

- **Zero grounding-seeking behavior.** Qwen3.5-9B asks a clarifying question on 0.00% of ambiguous prompts and 0.08% of non-ambiguous prompts. The delta between splits is statistically indistinguishable from zero. The Shaikh framework predicts a well-grounded model would show a meaningful delta here. Qwen shows **none**.
- **Refusal rates are symmetric (~2.3% vs ~2.5%).** The model refuses on both splits at similar rates — these are likely content-policy refusals, not grounding-driven "I need more info to answer" refusals.
- **Massive right-censoring.** 75-85% of responses hit the tokens_out ≥ 255 cap, meaning the harness captured an artificially truncated view of the true response distribution. See §2.6.
- **The 1 non-ambiguous clarifying question** is a curious outlier — worth inspecting in §2.5 qualitative samples (deferred to the labeler pass).

**Absence of assume-and-proceed metric:** I had planned to compute this as "response > 20 tokens AND no clarifying question in first sentence" but with 75% truncation this metric collapses to "response is long (always true for these long outputs) AND no question mark (always true for Qwen)". It's effectively ~100% on both splits and doesn't add information.

### 2.6 Right-censored tokens_out — methodological flag

**The `max_tokens=256` harness setting suppresses the true response-length distribution.** Post-run analysis:

```
ambig tokens_out buckets:
  255 (capped): 328  ← 75.4% of responses
  200-254:       7
  100-199:      51
  50-99:        26
  10-49:        22
  0-9:           1

non-amb tokens_out buckets:
  255 (capped): 1092  ← 84.5% of responses
  200-254:       41
  100-199:       79
  50-99:         51
  10-49:         25
  0-9:            4
```

**1420 of 1727 successful responses (82.2%)** hit the 255-token ceiling. This means:

1. The captured `tokens_out` distribution is **right-censored at 255** — the true distribution has more mass above 255 but the harness cut it off
2. Any analysis of response length (chars, tokens, structure) is biased toward "responses are exactly 255 tokens" when the reality is "responses are ≥255 tokens"
3. The proxy "response-is-a-question rate" may be underreported: if a question mark appears AFTER token 255 but before the true response end, we miss it. Unlikely for grounding-seeking (the clarification-ask should come FIRST in a well-grounded response), but possible
4. The binding "mean response length in chars" (920.9 / 1003.0) is close to what `255 tokens × ~4 chars/token = 1020` would predict — another symptom of uniform truncation
5. Latency measurements ARE accurate (they're wall-clock for the HTTP call, not content-dependent) but the model was doing MORE work than the tokens_out suggests

**Recommendation for future runs:** raise `max_tokens` to 1024 or higher (the harness `DEFAULT_MAX_TOKENS=256` at `scripts/run_rifts_benchmark.py:42`). Qwen3.5-9B's production config has `max_seq_len=4096` + chat template overhead ~50 tokens + worst-case prompt ~1900 tokens → up to ~2146 tokens of response budget without overflowing. Propose #248 as a harness patch.

**Why the 256 cap existed in the current run:** the harness `DEFAULT_MAX_TOKENS = 256` is a conservative default meant for smoke-test runs. The operator-triggered run used the default without adjusting for the production max_seq_len envelope. Not a bug — just an ergonomics gap.

**Template thesis:** A good benchmark predicts live behavior. A poor benchmark measures something systematically different from production. This section enumerates the divergence vectors hapax can test.

### 4.2 System prompt divergence

- **RIFTS harness sends no system prompt.** The chat completion is a single user-role turn with just the RIFTS `instruction` field.
- **Live voice loop attaches a ~2000-token persona system prompt** (`agents/hapax_daimonion/persona.system_prompt`).
- **Implication:** the RIFTS baseline measures Qwen3.5-9B's raw grounding behavior, not Qwen3.5-9B + hapax-persona. The persona system prompt may bias responses toward more/fewer clarifying questions in a way RIFTS cannot detect.
- **Compounding concern:** the production `max_seq_len: 4096` + ~2000-token persona leaves only ~2000 tokens for operator turn + response. Any live turn exceeding that budget gets truncated. See queue #229 closure inflection (2026-04-15T20:11:30Z) + proposed #231 for the raise-to-16K+ recommendation.

### 4.3 Multi-turn divergence

- **RIFTS is single-turn.** Each prompt is an isolated chat completion.
- **Live voice loop is multi-turn.** Grounding acts happen over N turns; RIFTS cannot measure the "operator clarifies, model proceeds on second turn" path.
- **Implication:** RIFTS may underestimate hapax's effective grounding because it doesn't give the model the conversational context that unlocks clarification-then-answer sequences.

### 4.4 Temperature divergence

- **RIFTS harness: temperature=0.0** (deterministic, for reproducibility)
- **Live voice loop: per-route temperatures** (~0.7 for conversational, 0.1 for coding/reasoning)
- **Implication:** live responses are more stochastic; RIFTS baseline is a lower bound on response-variance.

### 4.5 Thinking-mode divergence

- **RIFTS harness: thinking mode disabled** (v2 §2.1 E1: local-fast has thinking off in production, and the harness mirrors production)
- **Paper's frontier baselines presumably included thinking mode** (the paper is published before Qwen3.5 thinking was widely toggled off in production deployments)
- **Implication:** hapax's baseline is NOT directly comparable to the paper's 23.23% frontier number on a thinking-mode-apples-to-thinking-mode-apples basis. The comparison is "hapax production config" vs "paper frontier config". Document this explicitly; do not treat the 23.23% number as a same-substrate comparison target.

## 5. Recommendations for substrate decision

**Decision criteria** (from `docs/research/2026-04-15-substrate-reeval-post-hermes.md` §3):

1. Grounding behavior — does the model ask for clarification on ambiguous prompts?
2. Structured-output reliability — does pydantic-ai output_type parse cleanly?
3. Persona adherence — does the model hold the hapax voice?
4. Latency budget — does p95 fit the voice-loop envelope (<3s end-to-end)?
5. Tool use — does structured tool invocation work?
6. Post-training recipe transparency — is SFT-vs-DPO-vs-RL provenance documentable?

**RIFTS baseline informs criterion 1 directly + criterion 4 indirectly (latency).** The other four criteria need separate measurement.

### 5.1 Scenario 1 (keep Qwen3.5-9B only)

**Verdict: REJECT as standalone grounding substrate. KEEP for non-grounding workloads (structured output, tool use, persona, coding).**

**Rejection reasons:**

- **Zero clarifying-question rate on ambiguous split** (0 of 435) — Qwen does not perform grounding acts under production config. The third-strongest proxy signal in the Shaikh framework is absent.
- **No meaningful ambiguous/non-ambiguous delta** (0.00% vs 0.08%) — the model treats both splits identically. A grounding substrate should SHIFT its response shape based on prompt clarity; Qwen does not.
- **82.2% right-censored response length** confounds the qualitative analysis but does not rescue the clarifying-question signal (questions should come FIRST in a grounded response, not buried at token 300+).

**Preservation reasons (Qwen stays for non-grounding workloads):**

- 99.25% response rate + 0.75% transient error rate — production-reliable under the current config
- p50 latency 4.3s + p95 19s — within the voice-loop envelope for most turns (with stimmung downgrade for the p95 tail)
- Persona adherence + structured output + tool use validated independently of this report — none of those paths require grounding behavior

**Net:** Qwen3.5-9B continues as the default `local-fast`/`coding`/`reasoning` substrate for non-grounding traffic; a SECOND substrate is required for the claim-shaikh experimental arm.

### 5.2 Scenario 2 (parallel deploy OLMo 3-7B as second substrate)

**Verdict: DEPLOY (per drop #62 §17 Option C pivot ratification — parallel TabbyAPI :5001).**

**Rationale:** The §5.1 rejection of Qwen as a grounding substrate makes scenario 2 mandatory, not optional. OLMo 3-7B has documented SFT-only post-training with no RLHF mode collapse (beta substrate research v1 §4 + v2 §3), Apache 2.0 license, and fits the VRAM budget alongside Qwen3.5-9B. Cross-references to queue #211 (OLMo parallel deploy spec) and queue #212 (OLMo LiteLLM routes + claim-shaikh cycle 2 run).

**This baseline does change the urgency:** scenario 2 moves from "nice-to-have, parallel-deploy when convenient" to "required for the claim-shaikh arm to produce meaningful data". Queue #211 priority should be reviewed.

### 5.3 Scenario 3 (fall back to frontier Claude/Gemini for reasoning tier)

**Verdict: NO-OP (not currently viable given Anthropic credit balance + Claude fallback already in use).**

**Rationale:** The queue #229 audit found that LiteLLM is already falling back to Claude Sonnet (balance depleted per queue #232 operator item) and then to Gemini Flash for reasoning-tier calls. Adding a "route ambiguous prompts to Claude" rule would compound the credit pressure without improving the situation: the FRONTIER paper baseline is 2.22% accuracy on ambiguous split, meaning even Claude does not perform grounding acts at scale. The right fix is scenario 2 (parallel OLMo), not scenario 3 (more frontier traffic).
