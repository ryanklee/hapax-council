# Barge-In as Grounding Repair: Research and Design Direction

## The Problem

When the operator speaks during Hapax's response, the system goes deaf.
The speaking gate (`set_speaking(True)`) drops all audio except 1.5s
pre-roll. VAD is explicitly disabled during TTS playback to prevent
echo false positives. The operator's speech — corrections, reactions,
new information — is lost.

This is a grounding blackout. The system cannot track acceptance or
rejection during its own speech. Every long response is a period where
mutual monitoring stops.

## Current System State

### What Exists

| Component | Status | Location |
|-----------|--------|----------|
| Barge-in flag | Exists, never set (VAD disabled during TTS) | conversation_buffer.py:185-186 |
| Streaming loop check | Polls `barge_in_detected`, breaks loop | conversation_pipeline.py:958-972 |
| Audio playback | PyAudio blocking write, no interrupt API | conversation_pipeline.py:1509-1522 |
| Echo canceller | Speexdsp AEC, ~60-70% effective on Yeti | echo_canceller.py:131-174 |
| Turn phases | 5 phases including overlap detection | cognitive_loop.py:45-52 |
| Conversation thread | Sequential strings, no temporal metadata | conversation_pipeline.py:371-377 |

### What's Missing

- VAD during TTS (disabled to prevent echo false positives)
- Audio capture during system speech
- Graceful response completion (current: hard cut or nothing)
- Concurrent speech transcription
- Thread entries with temporal/overlap annotations
- Barge-in urgency assessment

## Proposed Architecture: (a→b→c→d)

### (a) Barge-in Detection

Operator speech detected during Hapax's response. Not the current
echo-prone VAD, but a higher-confidence detector:

Options:
- High-threshold VAD (0.95+) after AEC processing — only fires on
  strong human speech that exceeds residual echo
- Speaker ID embedding match in real-time — distinguish operator voice
  from Hapax echo using pyannote embeddings
- Energy ratio: mic energy after AEC / reference energy — human speech
  has different spectral characteristics than echo residual

### (b) Hapax Finishes Thought

Instead of hard cut, Hapax completes to the next clause boundary. The
streaming loop already chunks by clause (`_CLAUSE_END` regex). On
barge-in signal:
- Complete current clause (already being spoken)
- Don't start new clauses
- Transition to listening

This should be tuned, not ruled. The salience/activation system should
govern how much to finish based on:
- Barge-in signal strength (soft overlap vs. hard interruption)
- Remaining content value (mid-key-point vs. trailing filler)
- Conversational pacing (rapid exchange → finish less, slow discussion
  → finish more)

### (c) Capture Operator Speech During (b)

During (b), the mic stream continues. We already have:
- AEC reference feed (Hapax's TTS PCM queued to echo canceller)
- The TTS text (generated by LLM, known exactly)
- Mic residual after AEC ≈ operator speech

A secondary capture buffer accumulates mic frames after AEC during (b).
When (b) completes, this buffer feeds to STT. The transcript represents
what the operator said while Hapax was finishing.

Key: we don't need to separate operator from Hapax in audio. We know
Hapax's text already (it came from the LLM). We only need to transcribe
what the OPERATOR said, and the AEC residual is our best signal for that.

### (d) Context Injection for Next Round

The captured transcript enters the conversation as a concurrent
annotation, not a sequential turn:

```
Thread entry: "operator_clause → hapax_clause"
Concurrent: "[during response: operator said 'wait that's not right']"
```

This extends the thread format to carry temporal relationships:
- Sequential: A said X, then B said Y
- Concurrent: A said X while B was saying Y

The meta structure (bands, temporal flow) already models overlapping
temporal processes. A concurrent thread entry is a natural extension —
it's a perception event tagged with "happened during turn N" rather
than "happened after turn N."

For the LLM's next round, the system prompt or message history includes
the concurrent annotation. The model sees: "While I was explaining X,
the operator said Y" — and can respond to BOTH its own point and the
operator's interjection.

## Relationship to Clark's Grounding Theory

Clark & Brennan (1991) describe repair as the process participants use
to detect and resolve problems of speaking, hearing, and understanding:

- **Other-initiated repair**: the listener signals a problem. Barge-in
  IS other-initiated repair — the operator signals "wait" or "that's
  wrong" or "I need to add something."
- **Self-repair**: the speaker corrects themselves. Hapax finishing its
  thought gracefully (b) is a form of self-repair — completing the
  presentation unit before yielding.
- **Mutual monitoring**: participants monitor what their partners are
  doing moment by moment. Capturing operator speech during (b) IS
  mutual monitoring — the system hears the repair attempt while
  completing its own contribution.
- **Grounding record**: the repair enters the shared understanding.
  Context injection (d) preserves the repair as part of the
  conversation's grounding history.

The industry treats barge-in as binary: detect → cut. Profile-retrieval
systems discard the overlap entirely. Context anchoring treats barge-in
as a grounding signal carrying epistemic content that must be preserved
and integrated.

## Research Landscape (2025-2026)

### Full-Duplex Models

The frontier is end-to-end models that process both streams simultaneously:

- **NVIDIA PersonaPlex-7B** (Jan 2026): Moshi-based, 7B params, supports
  natural interruptions and overlapping speech. Genuinely concurrent
  listening and speaking in a single neural model.
- **Qwen2.5-Omni**: Thinker (multimodal LLM for reasoning) + Talker
  (dual-track autoregressive for streaming audio). Separation of
  reasoning from speech production.
- **Moshi**: Models user and system audio on parallel streams in the
  same underlying model.

These are architecturally different from our cascaded pipeline. They
don't have explicit barge-in handling because the model inherently
processes both streams. But they also don't have explicit grounding
repair — the overlap is implicit in the model's hidden state.

### Cascaded Approaches

Closer to our architecture:

- **FSM-based turn management**: SPEAK/LISTEN/IDLE states with explicit
  transitions. Control tokens (`<|S-S|>`, `<|S-L|>`) for turn switching.
- **FireRedChat**: Semi-cascaded with pluggable components. Some paths
  end-to-end, others cascade.
- **Full-Duplex-Bench** (Mar 2025): Benchmark for turn-taking capabilities
  including First Token Emission Delay, turn-switching F1, takeover rate.

### Evaluation Metrics

- Barge-In Detection Latency: time from user onset to TTS suppression
- True/False Barge-In Rates: correct interruptions vs. spurious stops
- Endpointing Latency: time to ASR finalization after user stops
- Turn-switching F1: accuracy of who-speaks-next prediction

### Gap

Nobody has a grounding-aware barge-in repair mechanism in a live voice
system. Full-duplex models process concurrent streams but don't preserve
the overlap as structured grounding data. Cascaded systems treat barge-in
as binary cut/resume. The (a→b→c→d) model is original.

## Implementation Components (Existing)

| Need | Existing Component | Extension Required |
|------|-------------------|-------------------|
| Operator/Hapax separation | AEC reference feed + known TTS text | Use AEC residual as operator signal |
| Clause-boundary finish | `_CLAUSE_END` regex in streaming loop | Stop after current clause on barge-in |
| Audio capture during speech | Pre-roll buffer (1.5s circular) | Secondary capture buffer during (b) |
| STT of captured audio | `resident_stt.transcribe()` | Feed secondary buffer post-(b) |
| Turn phase tracking | Cognitive loop 5-phase model | Add BARGE_IN_REPAIR phase |
| Thread entries | Sequential `"user → hapax"` strings | Add concurrent temporal tag |
| Urgency assessment | Salience router activation scoring | Run on barge-in transcript |
| Temporal metadata | Bands / meta-tempo in voice system | Concurrent annotations on thread |

## Decision Required After Baseline

Same as tool calls: after baseline establishes grounding works in the
simple turn-taking model, decide:

**(1a) Not needed for experiment**: If baseline grounding metrics are
strong with simple turn-taking, barge-in repair is a quality-of-life
improvement, not a grounding requirement. Implement as a post-experiment
enhancement.

**(1b) Needed for experiment**: If baseline reveals that long responses
consistently break grounding (frustration spikes, acceptance drops,
operator gives up), then the lack of barge-in repair is a confound.
The experiment would be measuring a system that goes deaf during its
own speech, not a system that maintains mutual monitoring.
